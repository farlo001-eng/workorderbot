WorkOrderBot — Bugfix Log
==========================
Date: 2026-06-18

FILE MODIFIED
-------------
  harvest_sync.py — parse_date() (around line 382)

CHANGE
------
  Fixed a ValueError crash caused by pandas NaT (Not a Time) values in date
  columns. NaT is a subclass of pd.Timestamp, but pd.isna(NaT) is True. The
  original code checked for float NaN before checking pd.Timestamp, so a NaT
  value fell through to the pd.Timestamp branch and crashed when .strftime()
  was called on it. The fix adds a pd.isna() check inside the pd.Timestamp
  branch, returning None for NaT before attempting to format it.

  harvest_sync.py — download_open_wo() and parse_date()

CHANGES
-------
  1. Added "--headless=new" and "--window-size=1920,1080" to the Chrome options
     in download_open_wo() so the Yardi login/download runs without a visible
     browser window.
  2. Removed the browser.maximize_window() call right after page load. In
     headless mode that call could override the explicit --window-size and
     shrink the viewport, which shifted the login page layout and caused
     Selenium to intercept clicks on the login button
     (ElementClickInterceptedException). The login button click now uses
     WebDriverWait + EC.element_to_be_clickable instead of an immediate
     find_element().click().
  3. parse_date() still crashed on pandas NaT after the earlier fix, because
     pd.NaT is a subclass of Python's datetime — not of pd.Timestamp — so it
     fell through the pd.Timestamp branch into the plain datetime branch, which
     called strftime() unconditionally. Merged the pd.Timestamp and datetime
     branches into one isinstance check with a single pd.isna() guard before
     formatting.

ASSUMPTIONS & NOTES

   values as "open" for tab membership, badge class, and label lookup. Raw
   Yardi statuses "web" and "call" keep their own badge color/label ("Web",
   "Call") but count as "not complete" for the All/Mine tabs, per the spec's

2. pendingStatus is initialized to the work order's raw status string (not
   normalized) when opening the detail view, so saving without touching the
   status buttons preserves whatever Yardi originally sent (e.g. "web")
   instead of silently rewriting it to "open". The three status buttons only
   normalize for the purpose of deciding which one to visually highlight.
   Clicking a button sets pendingStatus to that exact value going forward.

   existing backend behavior (app.py's COALESCE(ai_priority, 99) in
   /api/workorders): priorityNumber() collapses null and anything outside
   1-3 to 4 for card border color / filter chip grouping, while
   sortKeyPriority() treats null specifically as 99 so it still sorts after
   an explicit priority 4, matching the SQL ORDER BY.

   work order's property_code and the hardcoded TECH_PROPERTIES codes via
   parseFloat -> parseInt -> String, per logic note 1 (Yardi can return
   "79800.0").
5. Card click handling uses event delegation with a data-id attribute and an
   allWorkOrders lookup, rather than an inline onclick="openDetail(wo)" with
   a literal object — inline HTML can't embed a JS object literal safely, so
   this was a necessary implementation substitution for the same described
   behavior.
6. Photos are kept as a plain array on the in-memory work order object
   (currentWO.photos) after save, even though the API stores/returns them as
   a JSON string column — getExistingPhotos() handles both shapes.

   Screen" requirement described in the spec, and a service worker wasn't
